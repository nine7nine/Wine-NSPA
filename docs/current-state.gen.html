4. Wineserver global_lock PI – SHIPPED

Converted server/fd.c:global_lock from pthread_mutex_t to pi_mutex_t (FUTEX_LOCK_PI). This lock serializes all wineserver dispatch between the main epoll loop and per-client shmem dispatcher pthreads (NSPA v1.5). With PI, when a high-priority dispatcher (boosted by v2.4 client PI) contends with a lower-priority holder, the holder is automatically boosted via the kernel’s rt_mutex PI chain.

Background

Upstream wineserver is single-threaded – no locks. NSPA v1.5 added per-client shmem dispatcher pthreads, requiring global_lock to serialize state access. The original librtpi sweep excluded server/ because the audit assumed “single-threaded event loop = no contention.” That assumption was wrong – the v1.5 dispatchers create real contention, and without PI the lock was a priority inversion hazard when v2.4 client-side boost raised a dispatcher’s priority above the lock holder’s.

RT/PI Audit Finding

During this investigation, we validated that the server’s add_queue() / wake_up() priority ordering (originally planned as a separate change) is already handled by ntsync kernel patch 1002. With ntsync loaded, all sync primitives (events, mutexes, semaphores – both named and anonymous) go through the ntsync kernel driver for waiting, which provides priority-ordered waiter queues. The server-side add_queue() is never exercised for these objects. Confirmed by the wake-order test subcommand: 50/50 PASS regardless of server-side queue order.

7. Investigation Targets

1. PI contention variability (d4 vs d8/d12) [MEDIUM]

What we know: d4 RT PI avg is 238ms (8 samples, range 219-286ms). d8 and d12 show ~479ms avg (3 samples each, very tight range 477-481ms). The d4 run used 8 PI iterations, giving more opportunity for the holder to get CPU time. The d8/d12 runs only used 3 iterations.

What we don’t know: Is the d8/d12 higher avg caused by CFS load placement (4 busyloop threads competing for the SCHED_OTHER holder’s CPU), or by fewer samples masking variance? The d4 run also has a 67ms spread (219-286) suggesting CFS is the variable.

What would answer it: CPU pinning via taskset (pin holder + load threads to separate cores). Or increase d8/d12 iteration count to 8 to match d4.

Impact: Understanding, not a bug. PI boost is confirmed working at all depths.

2. Philosophers RT max wait 1620us [RESOLVED — PI v2]

Answer (2026-04-15): The cost was from buggy PI code, not sched_setattr_nocheck or list walk. The v3 ntsync_pi_recalc compared against owner->normal_prio which was already modified by sched_setattr_nocheck after a boost. This caused the comparison to fail on downward recalc, leading to unnecessary boost/unboost thrashing. PI v2 compares against saved orig_normal_prio, eliminating the thrashing. Result: 1620us → 865us (-46.6%).

3. Kernel mutex vs CS throughput gap [LOW]

What we know: ntsync rapid mutex: ~250K ops/s (kernel ioctl path). CS rapidmutex: ~301K ops/s (futex fast path with user-space spin). The ~20% gap is expected – CS has a user-space fast path (InterlockedCompareExchange on the lock word) that avoids the syscall entirely when uncontended. ntsync kernel mutex always takes an ioctl.

What we don’t know: Exact syscall overhead breakdown. Is it fixed ioctl dispatch cost, or does ntsync_pi_recalc add measurable overhead even in the uncontended case?

What would answer it: perf stat comparing ioctl enter/exit cycles vs futex enter/exit cycles. Or add an unlikely()-gated fast path in ntsync for uncontended mutex acquire.

Impact: Low – applications use CriticalSection for hot-path locking, not kernel mutexes. The kernel path is for cross-process sync which is inherently slower.

4. Wow64 rebuild [MEDIUM]

What we know: Last full rebuild was partial. 32-bit (i386) DLLs may be stale relative to the current 64-bit build.

What we don’t know: Whether any current test failures or behavior differences are caused by the stale 32-bit components. All current tests run as 64-bit PE binaries, so 32-bit issues would only surface with 32-bit Windows applications.

What would answer it: Clean configure --enable-archs=i386,x86_64 followed by make -j2 && make install prefix=/usr.

Impact: Required for 32-bit Windows application compatibility (many games, older DAW plugins).

5. ntsync module autoload [CRITICAL — RESOLVED]

What we learned: Without ntsync loaded at wineserver startup, get_inproc_device_fd() caches fd = -1 permanently. ALL sync waits serialize through wineserver global_lock. Under Ableton’s heavy threading, this causes priority inversion and deadlocks. This was NOT “convenience only” — it was a correctness issue.

Resolution: echo ntsync | sudo tee /etc/modules-load.d/ntsync.conf. Wine guardrail added in server/thread.c — loud CRITICAL warning when RT mode is active but /dev/ntsync is missing.

6. ntsync PI bugs (kernel driver) [HIGH — RESOLVED]

What we found (2026-04-15): Three PI bugs in the NSPA ntsync kernel patches:

(a) Multi-object PI corruption: Per-object orig_attr save/restore broke when a task held multiple mutexes being PI-boosted. Second mutex saved already-boosted scheduling as “original”. On first mutex unboost, owner dropped to SCHED_OTHER while second mutex still had RT waiters.

(b) wait_all had zero PI: ntsync_wait_all() added entries to obj->all_waiters but never called ntsync_pi_recalc(). The recalc function only scanned any_waiters. WaitAll with mutexes = no PI boost at all.

© Stale normal_prio comparison: After boost, sched_setattr_nocheck changed owner->normal_prio to reflect boosted state. When the highest-prio waiter left, highest_prio < owner->normal_prio evaluated false (both RT-level), so the boost was dropped entirely instead of being adjusted to remaining waiters' priority.

Resolution: Patch 1003 rewritten with per-device ntsync_pi_owner tracking (saves orig_attr once per task, restores only when ALL boosts removed), all_waiters scanning in PI recalc, and PI recalc calls in wait_all queue/dequeue paths. Module built, tested clean.
