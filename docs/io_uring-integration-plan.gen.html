Wine-NSPA io_uring Integration Plan

Date: 2026-04-14 (updated 2026-04-15) Status: Phase 1+2+3 COMMITTED, pool allocator COMMITTED, overlapped socket bypass WORKING

Background

rbernon attempted a full wineserver main-loop replacement with io_uring circa 2021-2022 (gitlab.winehq.org/rbernon/wine, branch archive/iouring). It was abandoned because io_uring was immature — missing features, kernel bugs, and API instability. The branch replaced epoll entirely and reworked request.c’s read_request/write_reply into io_uring linked SQEs.

Since then, io_uring (kernel 6.19) has gained: - IORING_SETUP_SINGLE_ISSUER + DEFER_TASKRUN (safe single-threaded rings) - IORING_OP_FUTEX_WAIT / FUTEX_WAKE / FUTEX_WAITV - Multishot poll, accept, recv - Provided buffer rings (io_uring_buf_ring) - IORING_OP_FIXED_FD_INSTALL (fd management from ring) - Zero-copy send/recv (SEND_ZC, SENDMSG_ZC) - IORING_SETUP_NO_SQARRAY (cache-efficient submission) - Ring-to-ring messaging (IORING_OP_MSG_RING) - 65 opcodes total, mature and stable API

Current NSPA Architecture (what we’re building on)

  • Shmem fast path (v1.5): request/reply via shared memory + futex, bypasses pipe I/O for the hot path entirely
  • PI futex (v2.3-2.5): FUTEX_LOCK_PI on global_lock, manual PI boost from client to server dispatcher thread (v2.5: cached sched state)
  • ntsync (/dev/ntsync): in-kernel event/mutex, bypasses server for sync primitive waits
  • Per-thread dispatcher pthreads: each client thread has a dedicated server-side dispatcher, RT-scheduled
  • epoll_pwait2: nanosecond-precision timeouts in wineserver main loop
  • SCHED_FIFO/RR mapping: NT priority bands → Linux RT scheduling

The shmem fast path already eliminated the biggest bottleneck (pipe I/O per request/reply). ntsync eliminated server round-trips for sync waits. io_uring therefore targets the REMAINING server-dependent paths.

OPPORTUNITIES — ranked by benefit/effort ratio

1. Client-Side Async File I/O (bypass server for NtReadFile/NtWriteFile)

Benefit: HIGH Difficulty: MODERATE Invasiveness: LOW-MED

What it does today: NtReadFile async path (file.c:6156) → register_async_file_read() → SERVER_START_REQ(register_async) → server adds fd to epoll → server detects readiness → alerts client → async_read_proc() → server_get_unix_fd() → non-blocking read() → completion

Every async file read/write requires: (a) A server round-trip to register the async (b) The server monitoring the fd via epoll © A server alert when ready (d) Another server_get_unix_fd() call in the callback (e) The actual non-blocking read/write

For RT audio workloads doing disk I/O (sample loading, streaming), this means the wineserver is in the critical path of every file op.

What io_uring gives us: Submit IORING_OP_READ/WRITE directly from the client thread. The kernel handles completion — no server involvement at all. When the CQE fires, we complete the NT async directly (set IOSB, signal event, queue APC).

This REMOVES the wineserver from the file I/O path entirely.

Implementation sketch: - Per-thread (or per-process) io_uring instance in ntdll - In register_async_file_read: instead of SERVER_START_REQ, submit IORING_OP_READ to the ring with the unix_fd and buffer - Completion: poll the CQ (or use eventfd notification) and call file_complete_async() directly - Fall back to server path for fds that can’t use io_uring (special server-side fds, console handles, etc.)

Key functions to modify: - register_async_file_read() file.c:5917 - async_read_proc() file.c:5586 - register_async_file_write() (similar pattern) - async_write_proc() file.c:5636

Interactions with NSPA: - Orthogonal to shmem fast path (shmem handles request/reply, this handles file I/O — different paths) - Reduces wineserver epoll load (fewer fds to monitor) - No conflict with PI or ntsync - Benefits RT: removes a server round-trip from the I/O critical path

Open questions: - fd lifetime: io_uring needs the fd to stay valid until CQE. The current model re-fetches fds via server_get_unix_fd() each callback. We’d need to hold the fd open for the duration of the io_uring op. Could use io_uring fixed fds + IOSQE_FIXED_FILE. - Cancellation: NT CancelIo → must cancel the in-flight SQE (IORING_OP_ASYNC_CANCEL). Need to wire this up. - Completion ports: IOCP notification must still work. After CQE, we’d call add_completion() ourselves — one server call, but only on completion, not on submission. - IOSB update from io_uring thread: must handle cross-thread IOSB writes carefully (STATUS_PENDING → final status).

2. Synchronous File I/O poll() Replacement

Benefit: MODERATE Difficulty: LOW Invasiveness: LOW

What it does today: NtReadFile synchronous path (file.c:6160-6190): poll(&pfd, 1, timeout) → loop back to read() NtWriteFile synchronous path (similar): poll(&pfd, 1, timeout) → loop back to write()

Each synchronous I/O that blocks does poll() + read()/write() — two syscalls minimum, sometimes looping.

What io_uring gives us: Single IORING_OP_READ/WRITE submission with the data buffer. The kernel handles the poll-wait internally. One syscall (io_uring_enter) instead of poll + read. For files that complete immediately, the SQE may complete inline with no kernel transition at all.

Implementation sketch: - In the synchronous blocking path, instead of: poll(fd, POLLIN, timeout) → read(fd, buf, len) Do: io_uring_prep_read(sqe, fd, buf, len, offset) io_uring_submit_and_wait(&ring, 1) // CQE has the result

Key functions to modify: - NtReadFile() synchronous wait file.c:6160-6190 - NtWriteFile() synchronous wait (similar)

Interactions with NSPA: - Zero interaction with shmem/PI/ntsync — purely client-side - Simplest possible change, good first PR

Caveat: - Need per-thread ring (or at least per-thread access to a shared ring). Ring setup cost is non-trivial — amortize by keeping it alive for the thread lifetime. - Timeout semantics: io_uring linked timeout (IORING_OP_LINK_TIMEOUT) can replicate the poll timeout behavior.

3. Socket I/O — recvmsg/sendmsg via io_uring

Benefit: MODERATE Difficulty: MODERATE Invasiveness: MODERATE

What it does today: try_recv() (socket.c:786) → recvmsg() try_send() (socket.c:1040) → sendmsg() async_recv_proc() → server_get_unix_fd() + try_recv() async_send_proc() → server_get_unix_fd() + try_send()

Same pattern as file I/O: server registers async, monitors fd via epoll, alerts client, client does non-blocking recvmsg/sendmsg.

What io_uring gives us: IORING_OP_RECVMSG / IORING_OP_SENDMSG — submit directly from client. Multishot recv (IORING_RECV_MULTISHOT) for streaming sockets: one SQE, multiple CQEs as data arrives. Provided buffer rings (io_uring_buf_ring) for zero-copy recv into pre-registered buffers.

Same server-bypass benefit as #1 — wineserver no longer monitors socket fds or dispatches alerts for socket I/O.

Implementation sketch: - In sock_recv(): instead of SERVER_START_REQ(recv_socket), submit IORING_OP_RECVMSG to the ring - For streaming sockets: use multishot recv + provided buffers - Completion: same as #1, poll CQ and complete NT async directly - Cancellation: IORING_OP_ASYNC_CANCEL

Interactions with NSPA: - Same as #1 — orthogonal to shmem/PI - Wineserver AFD poll monitoring becomes less loaded - Slightly more complex than file I/O due to msghdr/cmsg handling, address translation, MSG_OOB edge cases

Caveat: - Socket state machine is complex (connect, listen, accept, shutdown states). Must ensure io_uring submissions respect socket state. - Wine’s AFD implementation has careful error translation (sock_errno_to_status). Must preserve that in CQE handling.

4. Wineserver epoll → io_uring (main loop replacement)

Benefit: MODERATE Difficulty: HIGH Invasiveness: HIGH

What it does today: main_loop_epoll() (server/fd.c:606-668): - epoll_pwait2() for fd readiness - global_lock PI around the event dispatch - Processes up to 128 events per iteration

What io_uring gives us: - Linked operations: write reply + poll for next request in one SQE chain - Multishot poll: one SQE monitors an fd indefinitely (no re-arm) - IORING_SETUP_SINGLE_ISSUER + DEFER_TASKRUN: optimized for the wineserver’s single-thread-submitter model - Batch submission: fewer io_uring_enter() calls than epoll_wait()

This is what rbernon attempted. His approach: - Replaced main_loop_epoll with io_uring CQE loop - Added queue_fd_write/queue_fd_read/queue_fd_io for async I/O - Removed read_request/write_reply from request.c - Added io_event callback to fd_ops

Interactions with NSPA: - CONFLICTS with shmem fast path: shmem dispatchers bypass the main loop entirely. The main loop only handles fds that DON’T use shmem (fd passing, overflow, process management). - With shmem handling the hot path, the main loop is already NOT the bottleneck for RT workloads. - global_lock PI already covers the contention case. - Benefit is marginal given shmem exists.

Assessment: This was the right approach in 2021 when pipe I/O was the bottleneck. With shmem fast path in place, the wineserver main loop is no longer the critical path for request/reply. The complexity/risk of this refactor doesn’t justify the marginal gain.

SKIP THIS — unless #1/#2/#3 are done and we need more.

5. io_uring Futex Ops — Hybrid with Shmem Path

Benefit: LOW-MODERATE Difficulty: HIGH (research) Invasiveness: MED

What it does today: Shmem fast path: Client: atomic CAS + futex(FUTEX_WAKE) Server dispatcher: futex(FUTEX_WAIT) + acquire global_lock + dispatch Server: CAS + futex(FUTEX_WAKE) Client: futex(FUTEX_WAIT) + read reply from shmem

What io_uring gives us: IORING_OP_FUTEX_WAIT / FUTEX_WAKE — futex ops inside the ring. IORING_OP_FUTEX_WAITV — wait on multiple futexes simultaneously.

Possible use: - Server-side: instead of N dispatcher pthreads each blocked on futex(FUTEX_WAIT), use one io_uring with N FUTEX_WAITV SQEs. When any client’s futex fires, the CQE identifies which one. Reduces thread count on server side.

BUT: This conflicts with the per-thread dispatcher model, which exists specifically for PI (each dispatcher has its own RT priority that gets boosted by its client). Collapsing to a single ring would lose per-client PI isolation.

Alternative: Use IORING_OP_FUTEX_WAKE from the ring to chain “dispatch reply + wake client” atomically. Marginal gain at best — the futex WAKE syscall is already very fast.

Assessment: The shmem + futex path is already extremely tight. io_uring futex ops don’t offer meaningful improvement over raw futex syscalls for this use case. The PI model depends on per-thread dispatchers.

DEFER — not worth the complexity unless kernel gains PI-aware io_uring futex ops (which it doesn’t have today).

6. fd Passing — sendmsg/recvmsg with SCM_RIGHTS

Benefit: LOW Difficulty: LOW-MODERATE Invasiveness: LOW

What it does today: wine_server_receive_fd() uses recvmsg() with SCM_RIGHTS to receive file descriptors from the server. server_get_unix_fd() triggers this on cache miss (server.c:1367).

What io_uring gives us: IORING_OP_RECVMSG supports ancillary data (cmsg). Could submit the fd receive as an io_uring op, but SCM_RIGHTS fd passing is inherently synchronous from the application’s perspective — you need the fd before you can use it.

Assessment: Minimal benefit. The fd cache already avoids most round-trips. The uncached path is cold. Not worth the complexity.

SKIP.

7. Pipe Fallback Path (shmem overflow)

Benefit: LOW Difficulty: LOW Invasiveness: LOW

What it does today: When request/reply data exceeds 1 MB, the shmem path falls back to pipe-based I/O (the original Wine protocol).

What io_uring gives us: Could use IORING_OP_WRITEV/READV for the pipe fallback, batching header + data in a single submission.

Assessment: The overflow case is rare (1 MB is enormous for a wineserver request). Optimizing a cold path isn’t worth the complexity.

SKIP.

IMPLEMENTATION STRATEGY

Phase 1: Infrastructure + Synchronous File I/O (item #2)

Goal: Get io_uring plumbed into ntdll with minimal risk.

  1. Add configure.ac detection for liburing (borrow from rbernon)
  2. Add per-thread io_uring ring in ntdll unix layer
    • Initialize lazily on first file I/O
    • Ring size: 32 entries (small, grows if needed)
    • Flags: IORING_SETUP_SINGLE_ISSUER | IORING_SETUP_COOP_TASKRUN
  3. Replace poll() + read/write in NtReadFile/NtWriteFile synchronous paths with io_uring submission
  4. Benchmark: measure syscall count and latency for file-heavy workloads (sample loading, disk streaming)

Deliverable: Fewer syscalls for synchronous file I/O, no behavioral change, easy to A/B test.

Phase 2: Async File I/O Bypass (item #1)

Goal: Remove wineserver from the async file I/O critical path.

  1. New code path in register_async_file_read/write:
    • If fd is cached and io_uring available: submit directly to ring
    • Otherwise: fall back to server registration
  2. Completion handling:
    • Use eventfd registered with io_uring (IORING_REGISTER_EVENTFD)
    • eventfd fires → drain CQ → complete NT asyncs
    • Or: integrate CQ polling into ntdll’s wait infrastructure
  3. Cancellation: map NtCancelIoFile → IORING_OP_ASYNC_CANCEL
  4. IOCP: after CQE completion, call add_completion() (one server call on completion, not on submission — net reduction)

Deliverable: Async file I/O without server round-trips. Measurable wineserver CPU reduction under file I/O load.

Phase 3: Socket I/O (item #3) — COMPLETE

Goal: Same server-bypass for socket operations.

Implemented via io_uring POLL_ADD (not RECVMSG/SENDMSG directly): 1. sock_recv/sock_send: ALERTED block intercepts EAGAIN before set_async_direct_result. Sets E2 bitmap, submits POLL_ADD. 2. CQE fires → try_recv/try_send → set_async_direct_result(final) 3. Server async stays ALERTED (frozen), no epoll monitoring 4. ntsync uring_fd extension wakes sync waiters on CQE

Design choice: POLL_ADD + try_recv/try_send in CQE handler rather than RECVMSG/SENDMSG directly. This preserves Wine’s existing msghdr/cmsg/address translation in try_recv/try_send without duplicating it in io_uring-specific code. The poll overhead is negligible (one extra CQE vs direct completion).

Future optimization: multishot recv + provided buffers for streaming sockets. Current approach handles the common case well.

Deliverable: Socket I/O poll monitoring without server epoll. Phase A avg 95µs, Phase B (overlapped) avg 113µs.

RISKS AND CONSIDERATIONS

  • liburing dependency: Can vendor the header (rbernon did this) or require system liburing. System liburing is widely packaged now. For NSPA’s Arch packages, liburing is in [core].

  • Kernel version floor: SINGLE_ISSUER requires 6.0+, DEFER_TASKRUN requires 6.1+. NSPA is on 6.19, so no issue. For broader Wine upstream, would need graceful fallback.

  • fd lifetime: io_uring ops hold a reference to the fd for the duration of the operation. Must ensure the fd isn’t closed while an SQE is in flight. The current model of re-fetching fds per callback won’t work — need to hold fds open.

  • Thread safety: SINGLE_ISSUER ring can only be submitted to from one thread. Per-thread rings are the natural fit. If a shared ring is needed, would need IORING_SETUP_SQPOLL or mutex protection.

  • PI interaction: CONFIRMED SAFE. io_uring with SINGLE_ISSUER + COOP_TASKRUN processes completions in the submitting thread’s context (preserving RT priority). Kernel uses wait queues (not futexes) for io_uring_wait_cqe() — no conflict with FUTEX_LOCK_PI on other addresses. io_uring kernel workers only used for true async I/O (disk reads), not for completion delivery.

  • Memory pinning: io_uring may pin user pages for the duration of async I/O. For large buffers, this increases memory pressure. Monitor with io_uring_register_buffers if needed.

  • ntsync interaction: Confirmed orthogonal. io_uring uses kernel wait queues (not futexes) for CQE waits — no conflict with FUTEX_LOCK_PI. ntsync PI v2 (2026-04-15) fixed 3 kernel bugs independently of io_uring.

  • Shmem fast path interaction: None for phases 1-3. The shmem path handles request/reply IPC. io_uring handles file/socket I/O. They operate on different fd sets.

FINAL IMPLEMENTATION: E2 bitmap + ALERTED-state interception.

The overlapped bypass required 5 iterations to solve (2026-04-14 → 15):

Attempt 1 — CQ drain inline NtSetEvent: signal reentrancy crash.
Attempt 2 — Deferred completion flush: deadlock (event must signal
  DURING wait, not after).
Attempt 3 — Direct ntsync ioctl: double completion race (server
  re-queued async via set_async_direct_result(PENDING)).
Attempt 4 — Bitmap on sock + post-ALERTED bypass: ALERTED/PENDING
  race (bitmap set AFTER set_async_direct_result re-queued async).

Attempt 5 — ALERTED-state interception (WORKING):
  Intercept in the ALERTED block BEFORE set_async_direct_result.
  The async stays ALERTED on the server (terminated=1,
  async_waiting()=false → no epoll monitoring). The CQE handler
  calls set_async_direct_result once with the final result.
  The server accepts it because the ALERTED state flags
  (unknown_status=1, terminated=1, alerted=1) are all preserved.

  Key insight: set_async_direct_result(PENDING) restarts the async
  (terminated=0, alerted=0, async_reselect → server monitors via
  epoll). By never calling it, the async stays frozen in ALERTED.

Test results (v4-overlapped, 2026-04-15): Full suite: 20/20 PASS (10 tests × baseline + rt) Phase A (immediate): 2000/2000, avg 95µs, p99 162µs Phase B (overlapped): 2000/2000, avg 113µs, p99 189µs, 2000 async

Key finding: io_uring FUTEX_WAIT does NOT support FUTEX_LOCK_PI. The shmem manual PI boost (sched_setscheduler) cannot be replaced by io_uring futex ops. The producer-consumer signaling pattern doesn’t map to PI futex mutual exclusion semantics.

Global_lock benefit from E2: fewer socket fds in epoll → shorter main_loop_epoll iterations → shorter global_lock hold → less contention for shmem dispatchers. Indirect but real.

NOTE: We control the ntsync kernel module AND the Linux-NSPA kernel. ntsync could be extended with: - IORING_OP_URING_CMD handler → atomic wait-and-acquire from io_uring - PI-aware io_uring futex (futex2 PI flag + io_uring/futex.c) - Multishot poll support for ntsync fds These would unlock the full G vision (unified wait with PI).
