Shmem IPC PI Futex Redesign — REJECTED

Date: 2026-04-14 (updated 2026-04-15) Status: REJECTED — deadlocks on SMP. Manual boost (v2.5) is correct. Note: Implemented and tested 2026-04-15. Deadlocks due to FUTEX_LOCK_PI unlock/re-acquire race in dispatcher loop. See “Post-mortem” at end. The shmem PI v2.5 (cached sched state, 6→2 syscalls) remains the correct approach (commit 17621ba494c).

Overview

Replace the manual sched_setscheduler-based PI boost in the shmem IPC path with kernel-managed FUTEX_LOCK_PI. Eliminates a race window, removes ~60 lines of manual boost code, and provides atomic, correct priority inheritance through the kernel’s rt_mutex PI chain.

Current protocol (NSPA v2.4)

The shmem futex word is a plain signaling futex (0=idle, 1=request):

Dispatcher: Client: futex_wait(futex, 0) <-sleeping write request CAS futex 0->1 futex_wake wakes, sees futex==1 nspa_shm_pi_boost(): global_lock.lock() read dispatcher TID (atomic) dispatch request sched_getscheduler (syscall) write reply sched_getparam (syscall) CAS futex 1->0 sched_setscheduler (syscall) global_lock.unlock() futex_wait(futex, 1) <-sleeping futex_wake ———————–> wakes, reads reply nspa_shm_pi_unboost(): sched_setscheduler (syscall)

Race window: between TID read and sched_setscheduler, the dispatcher could change state (e.g. another client’s unboost lowering it). The window is small but not zero.

Manual boost: 3 syscalls (getscheduler + getparam + setscheduler). Manual unboost: 1 syscall. Total: 4 extra syscalls per request.

Proposed protocol: PI futex

Two futex words in shmem: pi_lock — FUTEX_LOCK_PI word (TID-based ownership, kernel rt_mutex) notify — plain futex for client->dispatcher wakeup

Dispatcher startup: futex_lock_pi(pi_lock) -> owns it (TID stored in futex word)

Dispatcher loop: Client: futex_wait(notify, 0) <-sleeping write request notify = 1 futex_wake(notify) wakes futex_lock_pi(pi_lock) <-blocks global_lock.lock() | kernel PI: boosts dispatcher dispatch request | ATOMIC — no race window write reply | transitive through global_lock global_lock.unlock() | futex_unlock_pi(pi_lock) ———-> wakes (now owns pi_lock) reads reply futex_unlock_pi(pi_lock) futex_lock_pi(pi_lock) <-reacquire goto loop

Why this works: 1. Dispatcher holds pi_lock while idle. Kernel tracks TID ownership. 2. Client’s futex_lock_pi blocks and atomically boosts dispatcher via kernel rt_mutex PI chain. No race window, no TID read, no sched_setscheduler. 3. PI is task-level: even though dispatcher sleeps on notify (different futex), the PI boost from pi_lock applies to the task_struct. When dispatcher wakes from notify, it runs at boosted priority. 4. Transitive PI: dispatcher (boosted) -> global_lock (pi_mutex) -> if contended, kernel boosts the global_lock holder too. Full chain. 5. futex_unlock_pi wakes client AND removes boost atomically. 6. Robust futex: if either side dies, FUTEX_OWNER_DIED is set. Other side can detect and handle gracefully.

Rebase impact

The changed files (server.c, thread.c, unix_private.h) already carry heavy NSPA patches. However, the change is a simplification — removing code and replacing one protocol with a cleaner one. The diff is smaller than the current boost code it replaces.

Concerns and mitigations

  1. futex_lock_pi is slightly more expensive than plain futex_wait (rt_mutex setup overhead). But we remove 4 sched_* syscalls per request, so net syscall count drops.

  2. Lock ordering: client acquires pi_lock, dispatcher holds global_lock. No circular dependency — pi_lock is always acquired by client, global_lock by dispatcher. The PI chain is linear: client -> pi_lock -> dispatcher -> global_lock -> holder.

  3. Teardown: current protocol uses futex=-1 for shutdown. New protocol needs equivalent — e.g. set notify=-1 and wake, dispatcher checks before re-acquiring pi_lock.

  4. Spurious wakeups: futex_wait(notify) can return spuriously. Dispatcher must re-check notify value, same as current futex==1 check. No change in structure.

  5. aarch64 barriers: futex_lock_pi/unlock_pi imply full memory barriers (they go through the kernel). The explicit __atomic_thread_fence calls around dispatch can be simplified or removed.

Relationship to io_uring work

This is independent of the io_uring integration. It improves the shmem IPC path regardless of whether io_uring is used for file/socket I/O. However, both reduce server-side overhead: - io_uring: removes file/socket fds from server epoll - PI futex: removes 4 syscalls per shmem request Together they compound — less kernel work per request, lower latency.

Future: kernel FUTEX_LOCK_PI in io_uring

io_uring’s IORING_OP_FUTEX_WAIT does NOT support PI (uses futex2 interface, no PI flag). Confirmed (2026-04-15): io_uring itself uses kernel wait queues for CQE waits, NOT futexes — so there is no conflict with FUTEX_LOCK_PI on other addresses.

A unified io_uring wait (sync objects + I/O + shmem PI in a single io_uring_enter) would require adding FUTEX_LOCK_PI support to the io_uring futex subsystem — a significant kernel change touching both io_uring/futex.c and core futex code. We control the NSPA kernel, so this remains a possible future extension, but is not currently planned.

Post-mortem: Why This Doesn’t Work (2026-04-15)

Implemented the full protocol (all 4 files changed, compiled clean) and tested. Deadlocks immediately on startup.

The bug — dispatcher unlock/re-acquire race on SMP:

After the dispatcher processes a request, the protocol requires: 1. Dispatcher: FUTEX_UNLOCK_PI(pi_lock) — wake client 2. Dispatcher: FUTEX_LOCK_PI(pi_lock) — re-acquire for next request

If the dispatcher is faster than the client (common on SMP — the dispatcher finishes processing before the client thread reaches FUTEX_LOCK_PI after the WAKE(notify) syscall returns):

1. Dispatcher UNLOCK_PI — no waiters (client hasn't blocked yet),
   kernel just clears the futex word to 0
2. Dispatcher LOCK_PI — re-acquires immediately (futex was 0)
3. Dispatcher WAIT(notify, 0) — sleeps, holding pi_lock
4. Client LOCK_PI(pi_lock) — blocks (dispatcher holds it)
5. DEADLOCK: client waits for pi_lock, dispatcher waits for notify.
   Reply is written but client can never read it.

Root cause: FUTEX_LOCK_PI can’t serve as BOTH reply notification AND PI mechanism. The unlock/re-acquire cycle has a window where ownership transfer to the client isn’t guaranteed — UNLOCK_PI only transfers atomically IF a waiter is already blocked.

Why manual sched_setscheduler works: it’s non-blocking. The PI boost is applied independently from the reply wait (FUTEX_WAIT on the state futex). The two mechanisms are decoupled.

Why this wasn’t caught in design review: the protocol is correct under the assumption that the client always reaches LOCK_PI before the dispatcher reaches UNLOCK_PI. This holds when the dispatcher does significant work (long request processing), but fails for fast/trivial requests where the dispatcher can complete the entire cycle before the client’s next syscall.

Conclusion: v2.5 manual boost (2 syscalls per RT request) is the correct approach for the shmem IPC protocol. FUTEX_LOCK_PI would require a kernel extension (e.g. combined notify+PI atomic operation) to work here.
