Sync Primitives Research: SRW Spin, Adaptive CS, Condvar PI

Date: 2026-04-15 Status: B (SRW spin) and C1 (pi_cond requeue-PI) IMPLEMENTED. A (CS DYNAMIC_SPIN) planned. C2 (SRW PI) deferred.

1. Windows Behavior (what apps expect)

SRW Locks

  • Spin count: ~1024 iterations hardcoded, using pause instruction
  • Spin on single CPU: Disabled (forced to 0)
  • Parking: NtWaitForAlertByThreadId (same as WaitOnAddress), per-thread futex
  • Bit layout (x64): bit0=exclusive_held, bit1=waking, bit2=multi_shared, bit3=shared_waiters, upper bits=shared_count OR wait_block_ptr
  • Wait blocks: Stack-allocated on waiter’s stack, linked list
  • PI support: NONE. Windows has no PI for SRW locks. Only kernel Mutant objects get PI.
  • Wake: NtAlertThreadByThreadId (single) or NtAlertMultipleThreadByThreadId (all)

Critical Sections

  • Default SpinCount: 0 (unless caller sets it)
  • RTL_CRITICAL_SECTION_FLAG_DYNAMIC_SPIN: System substitutes ~2000-4000, ignores caller’s value
  • Heap lock SpinCount: 4000
  • Spin loop: test LockCount → YieldProcessor() → decrement counter → repeat
  • Early bailout: if LockCount > 0 (multiple waiters), stop spinning
  • PI support: NONE on Windows. Our NSPA CS-PI via FUTEX_LOCK_PI is novel.

WaitOnAddress

  • No internal spin phase. Goes directly to kernel wait.
  • Callers (SRW locks) do their own spinning before calling WaitOnAddress.

Condition Variables

  • No PI. Pure user-mode, kernel doesn’t know what resource is waited on.

2. Linux/glibc Behavior (what our platform provides)

glibc Adaptive Mutex (PTHREAD_MUTEX_ADAPTIVE_NP)

  • Spin count: 100 iterations (fixed, MAX_ADAPTIVE_COUNT)
  • Algorithm: Simple loop — atomic_load_relaxed + _mm_pause(), decrement counter
  • No exponential backoff, no contention adaptation, no owner-on-CPU check
  • After 100 spins: Falls through to futex(FUTEX_WAIT)
  • Key weakness vs kernel: Spins blindly without knowing if owner is running

glibc pthread_rwlock

  • Spin count: ZERO. No spinning at all — direct futex(FUTEX_WAIT) on contention
  • Writer preference (NONRECURSIVE_NP): When writer waiting, new readers block
  • Uses two futex addresses (readers/writers) for selective waking
  • Note: PTHREAD_RWLOCK_PREFER_WRITER_NP is a no-op (identical to PREFER_READER)

Linux Kernel rwsem (CONFIG_RWSEM_SPIN_ON_OWNER)

  • The gold standard for rwlock spinning. Three-layer approach:
  • Layer 1 — Entry gate (rwsem_can_spin_on_owner):
    • Skip if need_resched(), NONSPINNABLE bit set, or owner NOT on CPU
    • Key insight: only spin if owner is actively running on a CPU core
  • Layer 2 — OSQ (Optimistic Spin Queue):
    • MCS-style queue lock serializes spinners — only ONE thread spins on the rwsem at a time
    • Others queue behind the OSQ head, spinning on their own local variable (cache-friendly)
  • Layer 3 — Owner-spin loop (rwsem_spin_on_owner):
    • No fixed iteration count — spins as long as same owner is on-CPU
    • After 1000 iterations without owner change, adds cpu_relax()
    • RT tasks get only ONE extra retry when owner is NULL
  • Reader-owned spin: Time-capped at (10 + nr_readers/2)us, max 25us
    • Checked every 16 iterations to avoid sched_clock() overhead
  • Handoff: After 4ms in wait queue, writer sets handoff bit (forced transfer)
    • RT/DL tasks set handoff immediately (no timeout)

Linux Kernel Mutex Adaptive Spin (CONFIG_MUTEX_SPIN_ON_OWNER)

  • Same OSQ + owner-on-CPU pattern as rwsem
  • Only spin while owner task is running on a CPU core
  • Falls to sleep when owner goes off-CPU or need_resched()
  • Unbounded iteration count (bounded by owner’s time on CPU)

PI for Read-Write Locks — State of the Art

  • Linux kernel: PI for rwlocks is explicitly unsupported
    • Documentation: “only a single owner may own a lock (i.e. no read-write lock support)”
    • Fundamental problem: PI requires single owner, rwlocks have N readers
    • Multi-reader PI is undefined for SCHED_DEADLINE (what bandwidth per reader?)
  • PREEMPT_RT approach: Rebuild rw_semaphore on rt_mutex
    • Writers acquire rt_mutex first (get PI from blocked readers)
    • Readers in slow path go through rt_mutex (acquire+release)
    • Asymmetry: Readers CAN boost waiting writer, but writer CANNOT boost multiple readers
    • Known limitation: low-priority reader can starve high-priority writer
  • No FUTEX_RDLOCK_PI or equivalent exists in Linux
  • Academic: RWPIP (Reader-Writer Priority Inheritance Protocol) requires O(N) boosts per writer block — never productively deployed
  • Practical answer for RT: Avoid rwlocks in RT-critical paths. Use PI mutexes, RCU, or seqlocks.

3. Wine Current Implementation

SRW Locks (dlls/ntdll/sync.c:852-967)

  • Spin count: ZERO. CAS loop → immediate RtlWaitOnAddress on contention
  • Bit layout: struct srw_lock { short exclusive_waiters; unsigned short owners; } = 4 bytes
    • exclusive_waiters: bit0 = held_exclusive, bits1-15 = waiter count
    • owners: shared owner count (0-65535)
  • Wait: RtlWaitOnAddress → user-space FIFO queue → NtWaitForAlertByThreadId → futex_wait
  • Wake: RtlWakeAddressSingle → NtAlertThreadByThreadId → futex_wake_one
  • No spin phase, no PI, no adaptive behavior

Critical Sections (dlls/ntdll/sync.c:703-747)

  • NSPA CS-PI path: nspa_cs_enter_pi() uses FUTEX_LOCK_PI (sync.c:341-409)
  • Static spin: Fixed YieldProcessor() loop with early bailout on LockCount > 0
  • DYNAMIC_SPIN flag: FIXME stub (sync.c:568)

RtlWaitOnAddress (dlls/ntdll/sync.c:1257-1386)

  • No spin phase. Compare-under-spinlock → add to FIFO queue → NtWaitForAlertByThreadId
  • 256-entry hash table of futex_queues indexed by addr >> 4
  • Fairness: FIFO (list_add_tail)
  • Indirect futex: Does NOT futex on the lock word — futexes on per-thread alert entry

Condition Variables (dlls/ntdll/sync.c:1092-1193)

  • Thin wrapper: release lock → RtlWaitOnAddress(condvar) → reacquire lock
  • No PI awareness. Wake is InterlockedIncrement + RtlWakeAddressSingle/All.

4. librtpi Capabilities

What it provides

  • pi_mutex_t: FUTEX_LOCK_PI-based mutex, user-space fast path via CAS on TID
  • pi_cond_t: Condition variable with PI mutex integration
  • No pi_rwlock_t. Reader-writer locks not supported anywhere in librtpi.

What’s integrated in Wine-NSPA

  • Tree-wide pthread_mutex → pi_mutex replacement (ntdll unix, audio drivers, gstreamer)
  • CS-PI via raw FUTEX_LOCK_PI in sync.c (PE-side critical sections)
  • pi_cond in audio drivers (alsamidi, jackmidi, pulse)

Critical gap: pi_cond doesn’t use requeue-PI

  • Wine-NSPA rtpi.h uses plain FUTEX_WAIT/FUTEX_WAKE for condvars
  • Upstream librtpi uses FUTEX_WAIT_REQUEUE_PI / FUTEX_CMP_REQUEUE_PI
  • Our gap: woken thread doesn’t get PI on mutex holder until explicit relock attempt
  • TODO comment at libs/librtpi/rtpi.h:30-31 acknowledges this
  • Impact: Window of priority inversion between condvar wake and mutex reacquire

6. Design Decisions

B — SRW Lock Spin Phase (RECOMMENDED FIRST)

Approach: Fixed bounded spin, matching Windows behavior.

RtlAcquireSRWLockExclusive: CAS attempt (existing fast path) if failed AND NumberOfProcessors > 1: for i = 0..SRW_SPIN_COUNT: read lock word if lock free: retry CAS if exclusive_waiters high: break early (contention) YieldProcessor() // pause instruction // fall through to existing RtlWaitOnAddress path

Spin count choice: ~256 iterations. - Windows uses ~1024 but we have higher syscall overhead (Wine→futex indirection) - glibc uses 100 but that’s for mutexes, not rwlocks - 256 is a reasonable middle ground — enough to catch short critical sections, bounded enough to avoid RT starvation - Can tune later with benchmarks

RT thread consideration: - On RT threads (detected via NSPA_RT_PRIO env or sched_getscheduler), use a SHORTER spin count (~32-64) to minimize holder starvation window, then fall to futex - This matches kernel rwsem behavior where RT tasks get limited retries

No per-instance state needed. Global constant + NumberOfProcessors check.

A — CS Adaptive Spin / DYNAMIC_SPIN

Approach: Implement the DYNAMIC_SPIN flag properly.

When RTL_CRITICAL_SECTION_FLAG_DYNAMIC_SPIN is passed: - Substitute system default spincount (~4000, matching Windows) - Store the flag in SpinCount high bit (existing pattern) - Optionally: adaptive adjustment based on acquire cost (port 0010 algorithm)

Gate behind !nspa_cs_pi_active() — when PI is active, the kernel handles contention better than userspace spinning.

C1 — pi_cond Requeue-PI Upgrade (RECOMMENDED SECOND)

Approach: Port upstream librtpi’s FUTEX_CMP_REQUEUE_PI usage.

This is a concrete, bounded fix for a known PI gap: - Replace FUTEX_WAIT with FUTEX_WAIT_REQUEUE_PI in pi_cond_wait - Replace FUTEX_WAKE with FUTEX_CMP_REQUEUE_PI in pi_cond_signal/broadcast - Waiters get atomically requeued to the PI mutex’s futex on wakeup - Eliminates the inversion window between wake and mutex reacquire

Constraint: FUTEX_WAIT_REQUEUE_PI requires the target mutex to be a PI futex (FUTEX_LOCK_PI format). Our pi_mutex already is — so this should work.

Constraint: This only helps code paths using pi_cond (audio drivers, unix-side mutexes). It does NOT help Windows-side condition variables (RtlSleepConditionVariableCS) because those use RtlWaitOnAddress, not pi_cond.

C2 — SRW PI (FUTURE RESEARCH)

Status: Unsolved problem. Not ready for implementation.

The fundamental issue: SRW locks have no owner. PI requires an owner. Options: - External hash table mapping lock address → owner TID (like RtlWaitOnAddress’s queue) - Adds overhead to every acquire/release (hash lookup + store) - Race conditions on reader tracking (multiple owners) - PREEMPT_RT approach: Serialize exclusive access through an rt_mutex behind the SRW - Only exclusive waiters get PI (readers don’t) - Changes lock semantics (acquisition order, fairness) - Accept the limitation: Windows itself has no SRW PI. Apps don’t expect it. - Focus PI effort on CS (already done) and condvar (C1) - For RT paths that need PI, apps should use CS not SRW

Recommendation: Don’t pursue SRW PI now. The effort-to-benefit ratio is poor given that PI for rwlocks is unsolved even in Linux. Instead, ensure CS-PI and condvar-PI are solid, and document that SRW locks are not PI-protected.

7. Implementation Status

  1. B — SRW spin phase – IMPLEMENTED (commit 005b55b4d8d)

    • 256-iteration spin in RtlAcquireSRWLockExclusive and RtlAcquireSRWLockShared
    • RT threads (NSPA_RT_PRIO active) skip spinning entirely
    • Disabled on single-CPU systems
    • Validated: 429 ntdll sync tests, 0 failures. RT suite 20/20 PASS.

  2. C1 — pi_cond requeue-PI – IMPLEMENTED (commit 43862d8b591)

    • FUTEX_WAIT_REQUEUE_PI / FUTEX_CMP_REQUEUE_PI in libs/librtpi/rtpi.h
    • Closes PI gap across all pi_cond consumers (ntdll, audio, gstreamer)
    • Benchmark: worst-case max 53.8us -> 31.6us (-41%) under RT load
    • RT suite 20/20 PASS.

  3. A — CS DYNAMIC_SPIN – Planned

    • Implement RTL_CRITICAL_SECTION_FLAG_DYNAMIC_SPIN (~4000 spincount)
    • Gate behind !nspa_cs_pi_active()
    • Low priority (PI path handles most contention)

  4. C2 — SRW PI – Deferred indefinitely

    • Unsolved problem, even Linux kernel doesn’t have rwlock PI
    • Documented as a known limitation
